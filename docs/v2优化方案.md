# VibeLive v2 优化方案

> 目标：把 VibeLive 从「一句话生成参数 + 氛围页面」升级为「一句话生成可听、可调、可解释的浏览器实时音乐作品」。

## 1. v2 目标

v1 已经跑通端到端链路：用户输入描述，AI 返回 `Vibe` JSON，浏览器实时合成音乐，并渲染视觉页面。v2.7 起音乐主引擎切换为 `@strudel/webaudio`，优先保证核心听感。

v2 的重点不是简单提高视觉复杂度，而是提升三件事：

- **音乐质量**：从随机音阶触发升级为有动机、段落、鼓型和低音线的结构化 loop。
- **生成可靠性**：从一次大 JSON 生成升级为分阶段生成、校验、修复。
- **可控体验**：把参数滑杆升级为更像 MIDI 控制器的声音操控面板，让用户知道自己在调什么。

## 2. 当前问题

### 2.1 音乐表达粒度偏粗

当前 `pattern` 只有：

- `root`
- `scale`
- `chords`
- `seed`
- `layers.{drums,bass,pad,melody,arp}`

这能控制基础氛围，但不足以表达真实音乐里的关键差异：

- 没有鼓组 step pattern。
- 没有 bassline。
- 没有 melody motif。
- 没有段落结构。
- 没有音色设计字段。
- 没有 variation / fill / breakdown。

### 2.2 AI 可能生成播放器不理解的和弦

当前调度器更适合罗马数字和弦，例如 `i / VI / III / VII`。但 AI 有时会生成 `Dm7 / G7 / Cmaj7` 这类绝对和弦名。播放器无法正确解析时，听感会退化为重复根级和弦。

### 2.3 生成等待缺少过程反馈

AI 生成可能需要 50-90 秒。当前 UI 只显示「正在生成」，用户不知道卡在哪一步，也不知道结果质量是否经过检查。

### 2.4 参数控制太像工程面板

`能量 / 温暖 / 空间 / 明亮 / 环境音 / 密度 / 速度` 本身是合理的底层参数，但原来的横向滑杆缺少音乐设备感。用户容易问「调这个有什么用」。

## 3. v2 设计原则

1. **先生成音乐蓝图，再生成视觉**  
   音乐质量优先。视觉服务于音乐，不抢先消耗模型上下文。

2. **让 AI 写谱，不是只写参数**  
   v2 的 AI 输出应包含可演奏的 pattern：鼓、低音、旋律、和弦、段落。

3. **播放器必须理解 AI 的语言**  
   schema 中允许的和弦、节奏、音符表达必须能稳定映射到 Strudel WebAudio pattern。

4. **控制器用音乐语言包装工程参数**  
   UI 可以保留底层参数，但呈现上要像硬件控制台：旋钮、小屏、键盘、状态灯。

5. **可解释、可修复、可局部再生成**  
   用户应该能知道生成结果的结构，也能只要求「鼓少一点」「旋律更明显」。

## 4. Strudel 对比与取舍

[Strudel](https://strudel.cc/workshop/first-sounds/) 是 live coding / pattern language。它的强项是用户可以直接写简短代码表达复杂音乐结构，例如鼓点、并行声部、子序列、采样 bank 和实时变化。

VibeLive 不应该变成要求用户写代码的 live-coding 工具。我们的定位是：

- 用户用自然语言描述氛围。
- AI 像 music director 一样写出可演奏 pattern。
- 播放器解释并演奏这些 pattern。
- 用户通过控制盘和自然语言继续调整结果。

因此，v2.7 起不再只“吸收 Strudel 思想”，而是把 `@strudel/webaudio` 作为主音乐 runtime。VibeLive 仍然不要求用户写代码：Strudel 被藏在 AI 与播放器之间，负责成熟 pattern、sample/synth 输出和 WebAudio 调度。

### 4.1 能力差异

| 维度 | Strudel | VibeLive v1 | VibeLive v2 目标 |
| --- | --- | --- | --- |
| 输入方式 | 用户写 live code | 用户写自然语言 | 用户写自然语言，AI 写谱 |
| 音乐表达 | mini-notation / pattern language | 参数 + layer density | AI 结构化乐谱 -> Strudel WebAudio pattern |
| 实时性 | 边写边听 | 生成后播放 | 生成后播放，可局部再生成 |
| 视觉 | 非核心 | 核心体验之一 | 音乐优先，视觉服务于音乐 |
| 适合人群 | live coder / 音乐技术用户 | 非专业用户 | 非专业用户 + 可解释音乐结构 |

### 4.2 v2.7 Strudel WebAudio 主引擎

当前工程决策：

- 引入 `@strudel/core`、`@strudel/mini`、`@strudel/webaudio`。
- 保留现有 `Vibe.pattern.mini` 作为 AI 输出兼容层。
- 播放时将 `drums / bassline / melodyMotif / arpPattern / chords` 翻译成 Strudel `stack(...)`。
- 使用 Strudel/superdough 的 sample 鼓、synth、room、delay、variation 和 scheduler。
- 代码预览展示真实 Strudel pattern，便于后续让 AI 直接输出 Strudel code。

兼容层仍支持：

```ts
interface MiniPatternV2 {
  drums: {
    kick: "x---x---x---x---";
    snare: "----x-------x---";
    hat: "x-x-x-x-x-x-x-x-";
  };
  bassline: "0---0---4---3---";
  melodyMotif: "0-2-4---2---0---";
}
```

约定：

- 字符串长度为 16 或 32，对应 16 分音符 step。
- `x` 表示触发鼓。
- `0-7` 表示音阶级数。
- `-` 表示休止。
- 大写或特殊语法暂不引入，避免 parser 过早复杂化。

这条路线先保证旧数据、分享链接、本地唱片架不失效，同时把核心声音输出换成成熟 WebAudio 音乐 runtime。当前 schema 已支持 `pattern.strudel.code`：AI 直接写受限原生 Strudel pattern，播放器安全校验通过后优先演奏；不安全或缺失时再回退到 `mini/chords/layers` adapter。

## 5. v2 数据模型建议

### 5.1 扩展后的 `VibePattern`

```ts
interface VibePatternV2 {
  version: 2;
  root: string;
  scale: ScaleType;
  bars: 8 | 16;
  seed: number;

  chords: ChordStep[];
  mini?: MiniPatternV2;
  drums: DrumPattern;
  bassline: NoteEvent[];
  melodyMotif: NoteEvent[];
  arpPattern?: NoteEvent[];
  arrangement: Arrangement;
  sound: SoundDesign;
}
```

### 5.2 和弦

```ts
interface ChordStep {
  degree: "I" | "ii" | "iii" | "IV" | "V" | "vi" | "vii" | "i" | "III" | "VI" | "VII";
  quality?: "triad" | "maj7" | "m7" | "sus2" | "sus4" | "add9";
  durationBars: 1 | 2 | 4;
}
```

要求：AI 只能输出罗马数字级数，禁止直接输出 `Dm7` 这类播放器不稳定解析的绝对和弦。

### 5.3 鼓组

```ts
interface DrumPattern {
  resolution: "16n";
  kick: number[];
  snare: number[];
  hat: number[];
  percussion?: number[];
  swing: number;
  humanize: number;
}
```

数组长度为 16 或 32，值域 `0-1` 表示力度。这样可以生成真正的节奏，而不是只用概率随机触发。

### 5.4 音符事件

```ts
interface NoteEvent {
  step: number;
  degree: number;
  octave: number;
  duration: "16n" | "8n" | "4n" | "2n";
  velocity: number;
  tie?: boolean;
}
```

`degree` 使用音阶级数，避免模型生成播放器不认识的音名。

### 5.5 编曲结构

```ts
interface Arrangement {
  introBars: number;
  loopBars: number;
  variationEveryBars: number;
  mutedLayersAtStart: LayerType[];
  fillSteps: number[];
}
```

v2 不再只是永远相同的 loop，而是同一主题的轻微变化。

### 5.6 音色设计

```ts
interface SoundDesign {
  padOsc: "sine" | "triangle" | "fatsaw" | "amsine";
  bassOsc: "sine" | "triangle" | "square";
  leadOsc: "sine" | "triangle" | "fatsine" | "pulse";
  texture: "vinyl" | "rain" | "wind" | "city" | "tape" | "none";
  filterMotion: number;
  saturation: number;
}
```

这会让「温暖」「明亮」「环境音」真正映射到听感，而不是只影响滤波和视觉。

## 6. 生成流程建议

### 6.1 v2 最小闭环：两阶段生成

第一阶段：生成音乐蓝图。

```txt
用户描述 -> 音乐导演 prompt -> MusicPlan JSON
```

第二阶段：根据音乐蓝图生成完整 vibe。

```txt
MusicPlan -> 视觉导演 prompt -> Vibe JSON
```

优点：

- 降低一次生成超时和跑偏概率。
- 让音乐质量优先。
- 方便单独重试音乐或视觉。

### 6.2 v2.5：轻量 agent 流程

后续可升级为 4 步 agent：

1. **Mood Director**：拆解用户描述，确定 genre、tempo、情绪、声场。
2. **Composer**：生成和弦、鼓型、低音线、旋律动机。
3. **Sound Designer**：生成音色、空间、环境纹理。
4. **Critic / Repair**：检查是否可循环、是否过密、字段是否合法。

这不是开放式 agent 工具调用，而是固定步骤的生成流水线，更稳定也更容易调试。

## 7. Prompt 方向

音乐生成 prompt 应从「生成参数」改成「生成可循环乐谱」。

核心约束：

```txt
你是浏览器实时合成音乐的作曲代理，不生成音频文件，只生成可循环演奏的结构化乐谱。

你必须先设计 16 小节循环：
- 4 小节 intro
- 8 小节主循环
- 4 小节变化或回落

输出必须包含：
- 和弦功能，只能使用罗马数字
- 鼓组 step pattern
- bassline step pattern
- 旋律 motif，必须有重复、留白和变化
- sound design 参数
- mix 建议

音乐规则：
- 旋律不能纯随机，必须围绕一个 3-5 音动机发展
- bass 要跟随和弦根音或五度
- 鼓组要有主拍、反拍或 ghost note
- 每 8 小节必须有一个可感知变化
- 必须适合无限循环
```

## 8. 前端体验优化

优先氛围感，目前不管是 ui 还是 icon、字体、布局 都比较工业化、廉价，需要有艺术感，并且要符合氛围感。

### 8.1 MIDI 控制盘

已开始将底部参数区从横向滑杆改为 MIDI 控制盘样式：

- 旋钮控制：能量、温暖、空间、明亮、环境音、密度、速度。
- 小屏显示：BPM、root、scale。
- mini keyboard：表达当前 loop 的音乐设备感。
- 保持原有参数数据流：`Slider -> onParamChange -> audio engine`。

后续可以继续增强：

- 旋钮拖动时显示参数解释。
- 给环境音增加真实音频纹理后，让控制盘显示 texture 类型。
- 当前播放 step 在 mini keyboard 上闪烁。

MIDI 默认收起不要影响氛围感；

### 8.2 生成过程可视化

把「正在生成」改成阶段进度：

1. 解析氛围
2. 作曲
3. 设计音色
4. 生成视觉
5. 检查循环

即使后端暂时仍是顺序请求，也能显著降低等待焦虑。

### 8.3 局部再生成

新增 prompt chip：

- 鼓少一点
- 旋律更明显
- 更低保真
- 更空旷
- 更适合专注

这些操作只更新 `pattern` 或 `params`，不必每次重做整个视觉。

## 9. 实施阶段

### Phase 1：修正 v1 音乐基础

- 限制 AI chords 只输出罗马数字。
- 调度器支持更多 chord quality。
- 环境音参数接入真实 texture layer。
- 保持 schema v1 兼容。

### Phase 2：引入 pattern v2

- 新增 `VibePatternV2` 类型与 schema。
- 先实现 mini-notation 子集 parser：鼓点 `x/-`，低音和旋律 `0-7/-`。
- 调度器支持 drums/bassline/melodyMotif。
- 内置 vibe 逐步迁移到 v2。
- 页面代码预览展示 v2 pattern。

### Phase 3：两阶段生成

- 第一片先升级现有单次生成：AI schema 必须输出 `pattern.mini`，播放器直接按 mini 演奏。
- `/api/generate` 内部拆成音乐蓝图和视觉生成，外部仍返回完整 `vibe`。
- 增加音乐蓝图 validator。
- 增加 repair prompt。
- 前端展示生成阶段。

### Phase 4：轻量 agent

- 固定 4 步音乐 agent 流水线：Mood Director、Composer、Sound Designer、Critic / Repair。
- 视觉生成作为音乐 agent 后置步骤，基于已校验音乐蓝图生成。
- 支持局部再生成：`/api/generate` 接收 `baseVibe` 与 `regenerateTarget`，播放器默认重编排音乐并保留当前视觉。
- 支持指定局部目标：music、drums、bass、melody、arp、visual。
- 支持根据用户反馈修正指定 layer。

### Phase 5：v2.7 Strudel WebAudio 主引擎

- 安装 `@strudel/core`、`@strudel/mini`、`@strudel/webaudio`、`@strudel/soundfonts`。
- 新增 `Vibe -> Strudel code` adapter，把现有结构化 pattern 翻译为 Strudel `stack(...)`。
- 音频主引擎从 Tone.js 调度器切到 Strudel scheduler + superdough WebAudio output。
- 首次播放加载 Strudel synths 和 `tidalcycles/dirt-samples` sample map；sample 加载失败时保留 synth fallback。
- Player 的代码面板展示真实 Strudel pattern。
- 保留原有播放、暂停、分享、保存、本地唱片架 UI 合约。

### Phase 6：AI 原生 Strudel Composer

- 在 `VibePattern` 中增加 `strudel` 字段，包含 `version/code/notes`。
- Composer 直接输出受限 Strudel pattern，而不是只输出 16/32 step mini 子集。
- 主生成链路使用 3 个 Composer 候选：旋律前景优先、groove 优先、质感空间优先；在进入 Sound Designer 前用本地质量评分选择最高分候选。
- Validator 检查 Strudel code 安全边界，播放器通过 `sanitizeStrudelCode` 与 `prepareStrudelCode` 兜底补齐 `.cpm(...)` 和 `.analyze(1)`。
- 新增本地音乐质量 gate：在 Critic 之后再次检查场景匹配、旋律前景、低音 drone、环境层/space/pad 过重、鼓组重心和整体密度；同时审查最终优先播放的 native Strudel 代码是否是多层 `stack(...)`、是否有 `note(...)` 前景、是否疑似单音 drone、空间/延迟是否过湿。出现 fail 级问题时做保守修复；spring/neon/space/focus/rain/forest/coastal/winter 会替换成手写 native Strudel repair recipe；如果这些已识别场景缺少 native `strudel.code`，也会自动注入对应 recipe。其他不明确场景才清空低质量 `strudel` 回退到结构化 adapter。
- `musicQuality` 会写入 Vibe，并随本地唱片架与分享链接保留；播放器展示质量分、修复状态和主要问题，便于后续定向再生成。
- 播放器提供定向反馈再生成：`太暗 / 太吵 / 旋律弱 / 不贴合 / 太空 / 太满` 会映射到明确的 `RegenerateTarget` 与追加 prompt，并把当前 `musicQuality.issues` 一起带入下一轮。
- 刻录日志会展示候选选择阶段，例如 `候选作曲 · 选中 2/3 · 92`。
- 新增 `npm run verify:music`：构造 8 类常见场景的坏 native/missing native Vibe，验证 quality gate 会注入对应 recipe、最终 Strudel 通过安全校验、没有残留 fail 级质量问题。

## 10. 成功标准

v2 不是看 JSON 多复杂，而是看用户是否能感受到：

- 每次生成的音乐有明确风格差异。
- loop 不再像随机音符堆叠。
- 调参能明显改变听感。
- 等待生成时知道系统正在做什么。
- 用户能用自然语言继续调整结果。
